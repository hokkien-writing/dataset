# AGENTS.md

## Project Overview

Hokkien/Teochew writing dataset project. External datasets (ChhoeTaigi, etc.) are imported, converted to unified CSV, merged into `merged.csv`, and exported as Rime input method dictionaries. Both Hokkien (POJ/TL/BP) and Teochew (PUJ/DP) romanization systems are supported.

## Build Commands

Full build pipeline (run from project root):
```bash
bash build.sh
```
This runs in order: `import_external.py` → `export.py` → `export_csv.py` → `merge_csv.py` → `export_rime.py`

Sync external datasets (clone repos, strip .git):
```bash
bash sync_external.sh
```

Deploy to Rime (macOS: `~/Library/Rime/`):
```bash
bash rime_deploy.sh
```

## Test Commands

Run all tests:

bash test.sh

## Code Style

### Python

- **Python 3.10+**: Uses `from __future__ import annotations`, `list[X]` type hints, `match` statements where applicable
- **No comments in code** unless explicitly requested
- **Imports**: stdlib → third-party → local. Use `from pathlib import Path` for file paths. Scripts add `PROJECT_ROOT = Path(__file__).resolve().parent.parent` and `sys.path.insert(0, str(PROJECT_ROOT))` to enable `from scripts.xxx import ...`
- **Dataclasses** for structured data (`ExternalEntry` in `importers/base.py`)
- **ABC + abstractmethod** for importer interfaces (`ExternalImporter`)
- **Naming**: `snake_case` for functions/variables, `PascalCase` for classes, `UPPER_SNAKE` for constants
- **Regex patterns**: Module-level compiled `re.compile()` constants prefixed with `_` (e.g., `_HAN_RE`)
- **CSV I/O**: Use `csv.DictReader`/`csv.DictWriter` with explicit `encoding="utf-8"`
- **File paths**: Always use `Path` objects, never raw strings
- **Error handling**: Minimal — let exceptions propagate. No defensive coding for impossible states
- **Functions**: Small, focused. Private helpers prefixed with `_`

### Bash

- **Header**: `#!/usr/bin/env bash` + `set -euo pipefail`
- **Directory**: `cd "$(dirname "$0")"` at script start
- **Quoting**: Always quote variable expansions in paths

### YAML (Rime schemas/dicts)

- Generated by Python scripts — do not hand-edit deployed files
- Dict files use `.dict.yaml` extension with Rime header format

## Architecture

### Pipeline Flow

```
external/ (raw data)           books/ lyrics/ (markdown with edit markers)
       ↓                                    ↓
import_external.py                      export.py → export_csv.py
       ↓                                    ↓
export/external/*.csv ─────────────→ merge_csv.py
                                            ↑
clippings/ (CSV) → export_csv.py → export/clippings/*.csv ──┘
                                            ↓
                                     export/merged.csv
                                            ↓
                                     export_rime.py
                                            ↓
                            export/rime/rime-hokkien/ + rime-teochew/
                                            ↓
                                     rime_deploy.sh → ~/Library/Rime/
```

### Key Directories

- `external/` — Raw files from external repos (no .git), tracked by `external/README.md`
- `export/` — All generated output (gitignored where appropriate)
- `export/merged.csv` — Wide table: `latn_norm, puj, dp, poj, tl, bp, han, han_variants, en, zh_CN, zh_TW, source`
- `export/rime/rime-hokkien/` — POJ, TL, BP schemas + dicts + lua filters
- `export/rime/rime-teochew/` — PUJ, DP schemas + dicts + lua filters
- `scripts/importers/` — One module per external dataset, inheriting `ExternalImporter`
- `scripts/latn/` — Romanization translators and converters (POJ↔TL↔BP↔PUJ↔DP)
- `scripts/processors/` — Book-specific CSV parsers + clippings processor

### Adding a New External Dataset

1. Add entry to `external/README.md` with source URL and license
2. Add clone line to `sync_external.sh`
3. Create `scripts/importers/<name>.py` extending `ExternalImporter`
4. Register in `scripts/import_external.py`'s `IMPORTERS` dict
5. Add tests in `scripts/tests/test_import_<name>.py`

### Adding New Clippings

1. Create `clippings/{teochew|hokkien}.{source}.csv` with header: `latn_norm,han,zh_TW,zh_CN,en`
2. File name prefix (`teochew`/`hokkien`) determines whether `latn_norm` converts to PUJ or POJ
3. `|`-separated variants in `latn_norm`, `han`, or `zh_TW` are auto-split into separate entries
4. `export_csv.py` processes clippings via `scripts/processors/clippings.py` → `export/clippings/`
5. `merge_csv.py` picks up `export/clippings/*.csv` automatically

## Important Conventions

- `export/external/*.csv` should NOT contain `latn_norm` — it's generated by `merge_csv.py`
- Annotations `(替)`, `(音)`, `(訓)`, `(白)` in POJ/TL → `[替]`, `[音]`, `[訓]`, `[白]` bracket notation during import
- Slash-separated variants (e.g. `bah-ke-á/bah-koe-á`) → split into separate entries
- Pipe-separated variants in clippings (e.g. `滂沛|霶霈` in `han`, `ta1-ke1|tua7-ke1` in `latn_norm`, `有條理|穩妥` in `zh_TW`) → split into separate entries with each combination
- Han-lo mixed entries (non-pure-hanzi HanLoTaibun) → set `han=""`, keep the entry
- `[...]` annotations are stripped when writing Rime dict files — they must not appear in `.dict.yaml` output
- `rime_deploy.sh` merges `rime.lua` and `default.custom.yaml` from both hokkien and teochew source dirs

## Project Skills

Skills in `.skills/` provide domain-specific guidance. Load them when relevant tasks arise.

### `.skills/rime-schema/`

**When to use:** Creating, modifying, or debugging Rime input schema files (`.schema.yaml`), dictionary files (`.dict.yaml`), customization patches (`.custom.yaml`), Lua scripts (`lua_filter`/`lua_translator`/`lua_processor`), or spelling algebra rules. Also when Rime deployment fails or candidates don't appear.

Key references in the skill:
- Schema skeleton, engine pipeline components (processors/segmentors/translators/filters)
- Dictionary format: tab-separated `text→code→weight` table after YAML header
- Spelling algebra operators: `xform` (destructive), `derive` (additive), `abbrev`, `fuzz`, `xlit`, `erase`
- Lua integration pattern: `rime.lua` loads modules, schema references via `lua_filter@name`
- Debugging checklist: YAML syntax, encoding, `schema_id` match, algebra order (longer patterns first), `alphabet` coverage

### `.skills/tone-mark-rules/`

**When to use:** Adding or fixing tone mark placement rules for a romanization system (PUJ, POJ, TL, etc.), configuring `tone_mark_priority` and `vowels` in a latn system config, or when `to_handwriting()` puts the diacritic on the wrong vowel.

Key concepts in the skill:
- Corpus analysis pattern: extract vowel/diacritic positions from reference CSV, separate vowel-initial vs consonant-initial syllables
- `tone_mark_priority` list: longest match first (composite entries before shorter substrings)
- `vowels` dict: simple vowels + composite entries for multi-vowel rhymes with special mark placement
- `vowel_initial_overrides`: when vowel-initial syllables behave differently (e.g., PUJ `ua` bare: consonant-init marks u, vowel-init marks a)
- `entering_tone_mark_before_ending=True`: for syllables ending in p/t/k/h, mark goes on vowel before ending
- `syllable_mappings` applied before tone matching (e.g., `ⁿ`→`nn`) so composite entries like `uaⁿ` match correctly
- PUJ reference rules derived from 8000+ corpus entries
